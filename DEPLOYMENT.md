## Step 1: Get Your Free API Keys

### 1a. Supabase Database
1. Go to [supabase.com](https://supabase.com) → Sign up free
2. Create a new project (name it `stock-picks`)
3. Go to **SQL Editor** → Run the schema below
4. Go to **Settings** → **API** → Copy your:
   - `Project URL` (looks like: `https://xyz.supabase.co`)
   - `Service Role Key` (keep this secret!)

**Database Schema** — Run this in Supabase SQL Editor:

```sql
-- Table: daily_picks (one row per day's screening)
CREATE TABLE daily_picks (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    run_date DATE NOT NULL UNIQUE,
    total_stocks_screened INTEGER DEFAULT 0,
    avg_score DECIMAL(5,2) DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Table: picks (individual stock picks per day)
CREATE TABLE picks (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    daily_pick_id BIGINT REFERENCES daily_picks(id) ON DELETE CASCADE,
    rank INTEGER NOT NULL,
    ticker VARCHAR(10) NOT NULL,
    company_name VARCHAR(255),
    current_price DECIMAL(10,2),
    previous_close DECIMAL(10,2),
    change_pct DECIMAL(6,2),
    volume BIGINT,
    avg_volume_90d BIGINT,
    market_cap BIGINT,
    sector VARCHAR(100),
    industry VARCHAR(100),
    pe_ratio DECIMAL(8,2),
    score DECIMAL(5,2),
    rsi DECIMAL(5,2),
    volume_ratio_20d DECIMAL(5,2),
    entry_suggestion TEXT,
    stop_loss DECIMAL(10,2),
    target_price DECIMAL(10,2),
    bullish_reasons TEXT[],
    risk_level VARCHAR(20),
    technical_score DECIMAL(5,2),
    fundamental_score DECIMAL(5,2),
    sentiment_score DECIMAL(5,2),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Indexes for fast queries
CREATE INDEX idx_picks_date ON picks(daily_pick_id);
CREATE INDEX idx_picks_ticker ON picks(ticker);
CREATE INDEX idx_picks_score ON picks(score DESC);

-- Enable Row Level Security (allow all reads for dashboard)
ALTER TABLE daily_picks ENABLE ROW LEVEL SECURITY;
ALTER TABLE picks ENABLE ROW LEVEL SECURITY;

-- Create policy to allow anonymous read (for Streamlit dashboard)
CREATE POLICY "Allow public read access" ON daily_picks FOR SELECT USING (true);
CREATE POLICY "Allow public read access" ON picks FOR SELECT USING (true);
```

## 🛠 Bypassing API Rate Limits

### yfinance Rate Limits
- **Problem:** yfinance from a single IP can get rate-limited.
- **Solution:** Our code already handles this with `time.sleep()` calls.
- **Additional protection:** We screen 500 stocks but only 150-200 typically pass initial filters.

### Finnhub (300/day)
- **Problem:** Only 300 API calls/day on free tier.
- **Solution:** We only fetch sentiment for the **top 15-20 candidates**, not all 500.
- That's only 20 API calls — well within limits.

### Supabase (Free Tier)
- **Problem:** 500MB database, 1GB bandwidth.
- **Solution:** 
  - We store only ~10 rows/day for picks (minuscule storage)
  - Each row is ~1KB → 365 days = 3.6MB (you have 500MB!)
  - Bandwidth: Dashboard reads ~50KB per load

## 🔧 Troubleshooting

### "No data available" on dashboard
1. Check GitHub Actions → Is the workflow running?
2. Check Supabase → Is there data in the `picks` table?
3. Try clicking **"Run Fresh Screen"** on the dashboard

### Pipeline fails with "ModuleNotFoundError"
- Make sure all dependencies are in `requirements.txt`
- Check that GitHub Actions is using the correct Python version

### API rate limiting errors
- The pipeline has built-in rate limiting (`time.sleep()` calls)
- If still hitting limits, try reducing the stock universe in `config.py`

## 📊 Understanding the Scoring Algorithm

### Formula: Total = Technical (65) + Fundamental (25) + Sentiment (10) = 100

**Technical Score (65 pts):**
| Component | Max Pts | How It's Calculated |
|-----------|---------|---------------------|
| RSI Momentum | 15 | RSI 50-70 = full points; >80 = penalized |
| MACD Signal | 15 | Fresh crossover = 15; in uptrend = 10 |
| Volume Spike | 15 | 2x+ avg = 15; 1.5x = 12 |
| MA Crossover | 10 | Golden cross = 10; above all MAs = 8 |
| Gap Potential | 10 | Near resistance + BB upper band = 10 |

**Fundamental Score (25 pts):**
| Component | Max Pts | How It's Calculated |
|-----------|---------|---------------------|
| P/E Ratio | 10 | < 15 = 10; < 25 = 7; > 60 = 0 |
| Market Cap | 5 | $100B+ = 5; $2B = 3 |
| Volume Ratio | 10 | High relative vol to mkt cap = 10 |

**Sentiment Score (10 pts):**
| Component | Max Pts | How It's Calculated |
|-----------|---------|---------------------|
| News Sentiment | 10 | Finnhub sentiment score × 10 |

### Scoring Logic (Pseudocode)
```
for each stock in universe:
    if price < $5: skip
    if market_cap < $500M: skip  
    if volume < 500K: skip
    
    fetch 60 days of OHLCV data
    compute RSI, MACD, Bollinger Bands, MAs
    score = 0
    
    # Technical
    if 50 <= RSI <= 70: score += 15
    if MACD_bullish_crossover: score += 15
    if volume_ratio > 1.5: score += 15
    
    # Fundamental
    if PE < 15: score += 10
    if market_cap > $10B: score += 4
    if volume / cap > threshold: score += 10
    
    rank by score
    take top 20, apply sentiment boost
    return top 10
```

## 📈 Making the Most of Free Tiers

### If you hit Finnhub's 300/day limit:
- Reduce sentiment analysis to only the **top 10 candidates**
- Edit `pipeline.py` → `get_batch_sentiment` call limit

### If you need more stocks in universe:
- Add more tickers to `config.py → DEFAULT_UNIVERSE`
- Note: yfinance will take longer (currently ~60 seconds)
- You can queue multiple GitHub Action runs if needed

### If Streamlit Cloud sleep timeout (after 7 days):
- Streamlit Cloud free tier: app goes to sleep after 7 days of inactivity
- Solution: Sign in to Streamlit Cloud and re-deploy, or use a free uptime monitor like UptimeRobot (free tier: checks every 30 min)
- Better solution: **Upstash Cron** (free) can hit your app URL every 30 min
